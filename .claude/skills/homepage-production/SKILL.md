# ホームページ制作の進め方（汎用・知見ベース）

新しいHP/LP制作の依頼が来たら**まずこれを読む**。目的は、過去に修正が集中した箇所を
**先回りで潰し**、少ない往復で仕上げること。以下は実案件（Revengeサイト）の会話ログを
定量分析して得た知見（修正回数の実測：スクロール系319・5行179・矢印114・ヒーロー91、
ヘッダー84・差し替え79、料金3・見た目不変系1）。

---

## 0. 鉄則（これだけは最初から）

1. **アニメーションが最大の修正源**。着手前に「動きの仕様」を言語化し、**数値を1箇所の定数**に
   出して即調整できる形にする。感覚調整は必ず往復するので“速く回せる”ことが正義。
2. **共有パーツ（ヘッダー/フッター/ナビ）は最初から単一ソース**にする。各ページに焼き込まない。
   後から「1箇所直すだけで全ページ反映」に作り替えると高コスト。
3. **キャッシュ更新用の `?v=` 等をHTMLに焼き込まない**。1ファイル変えると全ページが変わり、
   「全差し替え」地獄になる。更新はサーバの ETag/Last-Modified に任せる。
4. **納品は“変更に関係するファイルだけ”**で済む構造にする（差分最小デプロイ）。
5. **「見た目を変えるな」系の変更は、変更前後スクショで無変化を証明**してから出す。
6. 迷ったら **AskUserQuestion で選択肢化**して聞く（後述のキックオフ質問）。勝手に決めない。

---

## 1. キックオフで必ず詰める（往復を減らす質問）

依頼を受けたら、下記を**先に**確認する（実案件では後出しで各所が二転三転した）。

- **トーン/用語**：初心者向けにやさしく? 専門用語OK?（例：括弧注釈の有無まで決める）
- **料金表示**：載せる/伏せる? 固定額/範囲? 税込/税別? 一括/月額（保守）? 「後で」なら枠だけ作る。
- **メニュー/事業の数と増減**：今後増える? → 増えるなら**データ駆動**（配列から生成）にしておく。
  カラム数（3列/4列…）は件数で変わるので固定値にしない。
- **アニメーションの有無と体感**：ヒーローの演出は? どのくらいの“量/速さ/間”? 自動 or スクロール駆動?
  一時停止やロックは要る?（ここが最も割れる。仕様を文章で固定してから作る）
- **共有パーツ/更新運用**：更新は誰がどう反映? 「ヘッダーだけ直したい」時に何ファイル触る想定?
- **公開/ドメイン/ホスティング**：静的? PHP可? WordPress併用? 段階公開（審査待ち等）の制約は?
- **成長ロードマップ**：LP追加、外部システム埋め込み、多言語 等の予定。設計に織り込む。

---

## 2. 修正多発ポイントと対処（実測ランキング順）

### ① アニメーション／ヒーロー演出 ＝ 断トツで最多（要注意）
- **なぜ割れる**：「もう少し短く/長く」「量を多め/少なめ」「一時停止する/しない」など体感依存で、
  文章で合意しても実機で必ず調整が入る。実案件では同一挙動を10回以上調整（値が往復した）。
- **対処**：
  - 挙動を**単一の定数**に集約（例：1行あたりのスクロール量 `BIG_LEN` 等）。1値変更→即再生成で回す。
  - 変更のたびに**Playwrightで「+Npx→N行進む」等を数値で確認**し、体感を再現可能にする。
  - 「揃ったら一時停止」「流れ切るまで遷移ロック」等の**状態遷移は先に図/箇条書きで固定**してから実装。
  - 初期値は必ず「初期値です、実機で微調整します」と添えて出す（往復前提を共有）。
- **やりがちな失敗**：CSSの`prefers-reduced-motion`を無視して検証がブレる → スクショ検証時は
  reducedMotionを固定して決定的に。

### ② ヘッダー/ナビ ＆ 差し替え運用 ＝ 2番手（保守性）
- **なぜ割れる**：全ページに焼き込むと、ナビ1項目追加でも全HTML差し替えになり「面倒」と必ずなる。
- **どう直したか（実案件）**：ヘッダーの“正”を共有JS（全ページが読む1ファイル）に置き、起動時に
  各ページのヘッダーを最新版へ差し替える方式へ変更。焼き込みはJS無効/クローラ向けフォールバックに残す。
  → 以後ヘッダー変更は**そのJS1ファイルの差し替えだけ**で全ページ反映。
- **対処（新規案件）**：最初から共有パーツを単一ソース（サーバサイドinclude or 生成器 or 共有JS）に。
  相対パス接頭辞（サブフォルダ対応）と現在地ハイライトはJS/生成側で自動判定。

### ③ キャッシュバスター `?v=` の焼き込み
- **症状**：CSS/JSのハッシュを全HTMLに `?v=` で焼くと、アセット1つの変更で全HTMLが変わる＝全差し替え。
- **どう直したか**：`?v=` を廃止し、ETag/Last-Modified 再検証に委譲。**アセットを変えてもHTMLは不変**に。

### ④ 料金/パッケージ表示
- **実際は即決着（3回）**だが、初期に表示方針（載/伏・固定/範囲・税・月額）を聞けば0回にできた。
- パッケージが増える前提なら**データ駆動**にして、追加を最小差分に。

### ⑤ レイアウト（カラム数・横並び）
- 事業/カードの件数変更で 3列⇄4列 等が動く。**カラムはデータ件数から算出**、決め打ちしない。

### ⑥ 「ブラウザで見える変化は禁止」系（SEO/AIO・コード整理）
- head メタ/JSON-LD 追加やコード整理は**見た目不変が要件**。→ 変更前後のスクショ比較で無変化を証明して納品。

### ⑦ テンプレ流用時の名称置換
- テンプレ（別名の雛形）から作る場合、社名等は**一括置換関数**に集約（例 rebrand()）。手置換は漏れる。

## 3. 推奨アーキテクチャ（静的サイトの定石）

- **生成器（gen.py 等）でHTMLを出力**。共通の head/header/footer/hero/breadcrumb を関数化し、
  ページはデータ配列＋本文から組み立て。HTMLは直接編集しない（再生成で一貫性を担保）。
- **FLOCSS 等の命名規約**でCSSを構造化。JSは全ページ共通1本を末尾読み込み、要素存在チェックで各機能を起動。
- **共有ヘッダーは単一ソース**（上記②）。**`?v=`は使わない**（上記③）。
- SEO/AIO：robots.txt / sitemap.xml / llms.txt / JSON-LD を生成器から自動出力。
- ニュース等の更新頻度が高い要素は WordPress ヘッドレス等と**ハイブリッド**にし、静的側はフォールバック表示。

---

## 4. 検証の型（毎回）

```bash
python3 -m http.server 8123 --directory <repo>        # ローカル配信
NODE_PATH=/opt/node22/lib/node_modules node verify.js  # Playwright
#   launch { executablePath:'/opt/pw-browsers/chromium' }
#   決定的スクショは emulateMedia({ reducedMotion:'reduce' })
```

- 変更に応じた観点で確認（ナビ開閉/リンク/現在地、本文、アニメの数値挙動、レスポンシブ）。
- **見た目不変が要件なら before/after スクショ比較**。
- **ライブ確認**：ブラウザがポリシープロキシで実サイトに直接つなげない環境では、
  `curl`で実ファイルをローカルにミラー → http.server → Playwright で「実際に配信中の組合せ」を描画確認。

---

## 5. 納品・運用

- **変更に関係するファイルだけ**を同階層で上書き（全差し替えを避ける設計を①〜③で担保）。
- 差し替え履歴を残す（どのファイルを上げるか明記）。個別ファイルで受け渡し。
- コミット/ブランチ/ログ規約はプロジェクトの指定に従う（トレーラ、指定ブランチ、PRは依頼時のみ）。
- 外部送信/公開は不可逆。段階公開の制約（審査待ち等）があれば**公開ゲート**を設けてタスク管理。

---

## 6. アンチパターン（過去にハマった）

- ❌ アニメの値をCSS/JSに直書きで散らす → 調整のたびに探す。**定数集約**へ。
- ❌ ヘッダーを全ページ焼き込み → 1項目追加で全差し替え。**単一ソース**へ。
- ❌ `?v=` をHTMLに焼く → アセット変更で全HTMLが変わる。**廃止**。
- ❌ カラム数・事業数を決め打ち → 件数変更で崩れる。**データ駆動**へ。
- ❌ 「見た目不変」依頼を無検証で納品 → 事故。**スクショ比較**必須。
- ❌ 仕様を聞かず実装 → 二転三転。**AskUserQuestionで選択肢化**して先に確定。

---

## 7. 進め方チェックリスト

1. キックオフ質問（§1）で仕様確定 → 特にアニメ/料金/共有パーツ/成長性。
2. アーキテクチャ決定（§3）：生成器・共有ヘッダー単一ソース・`?v=`なし・データ駆動。
3. 実装（HTMLは生成器経由／アニメ値は定数集約）。
4. 検証（§4）：数値で挙動確認、見た目不変ならスクショ比較。
5. 納品（§5）：関係ファイルのみ、履歴記載。
6. 反復（特にアニメは往復前提。初期値と明記して速く回す）。
